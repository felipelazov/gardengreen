## Story

**As a** jardineiro,
**I want** que fotos sejam enviadas automaticamente quando houver internet,
**so that** nao me preocupe com upload manual.

## Acceptance Criteria

```gherkin
AC-1: Salvar foto offline
GIVEN uma foto tirada offline
WHEN o usuario tira a foto
THEN a foto e salva localmente imediatamente
AND fica disponivel no app (thumbnail)
AND icone indica "Aguardando upload"

AC-2: Upload automatico
GIVEN fotos pendentes de upload e conexao restabelecida
WHEN o background sync detecta conexao
THEN faz upload das fotos em fila (uma por vez)
AND comprime para max 1MB antes do upload
AND atualiza icone para "Enviado" apos upload completo

AC-3: Retry com backoff
GIVEN um upload falhando (conexao instavel)
WHEN o upload e interrompido
THEN o sistema retenta em 5 minutos
AND apos 3 falhas, exibe notificacao: "X fotos pendentes de envio"
```

### Implementacao

- [ ] **Task 3** (AC: 1): Implementar captura e salvamento local de fotos

  - Diretorio local: `FileSystem.documentDirectory + 'photos/'`
  - Nome do arquivo: `{uuid}.jpg`
  - Criar registro no WatermelonDB com `upload_status: 'pending'`
  - Retornar local_path para exibicao imediata do thumbnail
  - [Source: architecture.md#5.4, architecture.md#11.4]

- [ ] **Task 4** (AC: 2): Implementar logica de compressao de imagens
  - Path: `apps/mobile/src/services/photoService.ts` (funcao `compressPhoto`)
  - Usar `expo-image-manipulator` para redimensionar e comprimir
  - Regras: max 1080px na dimensao maior, JPEG qualidade 80%, resultado final max 1MB
  - Se imagem ja for <= 1MB, nao recomprimir (evitar degradacao desnecessaria)
  - Retornar path da imagem comprimida e `file_size` em bytes
  - [Source: architecture.md#11.4 Compressao]

  ```typescript
  // apps/mobile/src/services/photoService.ts
  export async function compressPhoto(localPath: string): Promise<{ path: string; size: number }> {
    const result = await ImageManipulator.manipulateAsync(
      localPath,
      [{ resize: { width: 1080 } }],
      { compress: 0.8, format: ImageManipulator.SaveFormat.JPEG }
    );
    const info = await FileSystem.getInfoAsync(result.uri, { size: true });
    return { path: result.uri, size: (info as any).size ?? 0 };
  }
  ```

- [ ] **Task 5** (AC: 2): Implementar upload para Supabase Storage

  - Apos falha: atualizar `upload_status` para `'failed'` e incrementar contador de tentativas

### Arquitetura Relevante

**Photo Model (WatermelonDB):**
- Tabela `photos` com campos: `id`, `service_id`, `client_id`, `user_id`, `type`, `url`, `local_path`, `upload_status`, `file_size`
- `upload_status` enum: `pending | uploading | uploaded | failed`
- Campo auxiliar `retry_count` (integer, default 0) para controle de tentativas
- [Source: architecture.md#10.4 — photos schema]

**Compressao:**
- Biblioteca: `expo-image-manipulator`
- Constraints: max 1080px na maior dimensao, JPEG qualidade 80%, resultado final max 1MB
- Nao recomprimir imagens que ja estao dentro do limite
- [Source: architecture.md#11.4]

**Supabase Storage:**
- Bucket: `garden-photos`
- Path no bucket: `{user_id}/{client_id}/{service_id}/{uuid}.jpg`
- RLS enforced via pasta `{user_id}/` — usuario so acessa suas proprias fotos
- URL publica disponivel apos upload bem-sucedido
- [Source: architecture.md#18.4 Storage, architecture.md#10.4]

**Upload Queue — Backoff Schedule:**
- Tentativa 1: imediata (ao detectar conexao)
- Tentativa 2: 5 minutos apos falha
- Tentativa 3: 15 minutos apos falha
- Tentativa 4: 60 minutos apos falha → notificar usuario
- Nao cancelar a tentativa manual do usuario (retry on demand deve bypassar backoff)
- [Source: architecture.md#11.4 — exponential backoff]

**Deteccao de Conectividade:**
- Usar `@react-native-community/netinfo` (`NetInfo.addEventListener`)
- Listener deve ficar ativo durante toda a sessao autenticada
- Iniciar processamento da fila imediatamente ao detectar `isConnected: true`
- [Source: architecture.md#11.4]
