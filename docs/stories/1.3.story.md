## Story

**As a** novo usuario,
**I want** configurar o app rapidamente e cadastrar meu primeiro cliente,
**so that** comece a usar imediatamente sem perder tempo.

## Acceptance Criteria

```gherkin
AC-1: Onboarding flow
GIVEN um usuario que acabou de se registrar
WHEN o onboarding inicia
THEN a tela 1 solicita: nome profissional (pre-preenchido se veio do Google)
AND a tela 2 solicita: cidade e tipo de servico principal (lista pre-definida)
AND a tela 3 convida: "Cadastre seu primeiro cliente" com formulario minimo

AC-2: Primeiro cliente
GIVEN o usuario na tela 3 do onboarding
WHEN ele cadastra nome e telefone do primeiro cliente
AND agenda o primeiro servico (data + tipo)
THEN o onboarding e concluido com animacao de parabens
AND o usuario e levado para a tela principal com seu servico visivel

AC-3: Pular onboarding
GIVEN o usuario querendo pular o onboarding
WHEN ele toca em "Pular" em qualquer tela
THEN o onboarding e encerrado
AND o usuario vai para a tela principal vazia
AND um tooltip sutil indica "Cadastre seu primeiro cliente"
```

[Source: PRD Section 8, EPIC-01, Story 1.3]

### Logica de Negocio

- [ ] **Task 8** (AC: 1, 2, 3): Implementar logica de conclusao do onboarding
  - Ao concluir cada step, salvar dados parciais (estado local com `useState` ou context de onboarding)
  - Ao concluir Step 1 + Step 2: chamar `authStore.updateProfile({ name, city, service_type })`
  - Ao concluir Step 3:
    1. Criar cliente via `clientStore` → salva em WatermelonDB local + enfileira sync com Supabase
    2. Criar servico via `serviceStore` com `client_id` do cliente recem-criado
    3. Atualizar `users.onboarding_completed = true` e `users.profile_completion = 60` via `authStore.updateProfile()`
    4. Atualizar `authStore.isOnboarded = true`
  - [Source: architecture.md#10.3 State Management — authStore.updateProfile, clientStore, serviceStore]

  ```typescript
  // Ordem de operacoes ao concluir onboarding (Step 3)
  const handleConcluir = async () => {
    // 1. Criar cliente local + sync queue
    const client = await clientStore.createClient({ name, phone });
    // 2. Criar servico local + sync queue
    await serviceStore.createService({ client_id: client.id, date, type });
    // 3. Marcar onboarding como concluido no Supabase
    await authStore.updateProfile({ onboarding_completed: true, profile_completion: 60 });
    // 4. Navegar para tela de parabens
    router.push('/(auth)/onboarding/congrats');
  };
  ```
  [Source: architecture.md#10.3 State Management, architecture.md#10.4 Offline Layer (WatermelonDB)]

### Arquitetura Relevante

**Navegacao de Onboarding:**
- Rota base: `apps/mobile/app/(auth)/onboarding/`
- Expo Router file-based: cada arquivo = uma rota
- Fluxo: `step1.tsx` → `step2.tsx` → `step3.tsx` → `congrats.tsx` → `(tabs)/agenda/`
- Alternativa de pulo: qualquer step → `(tabs)/agenda/`
[Source: architecture.md#10.1 Navegacao]

**Persistencia de Dados:**
- Cliente criado: WatermelonDB (local, imediato) + sync queue para Supabase
- Servico criado: WatermelonDB (local, imediato) + sync queue para Supabase
- Profile update (`onboarding_completed`, `city`, `service_type`): Supabase direto (requer conexao) + atualiza authStore local
- Se offline ao completar onboarding: enfileirar todas as operacoes no sync queue
[Source: architecture.md#10.4 Offline Layer (WatermelonDB)]

**Tabela `users` — campos atualizados pelo onboarding:**
```sql
-- Step 1
UPDATE users SET name = $1 WHERE id = $auth_user_id;

-- Step 2
UPDATE users SET city = $1, service_type = $2 WHERE id = $auth_user_id;

-- Conclusao (Step 3 ou Pular)
UPDATE users
SET onboarding_completed = TRUE,
    profile_completion = 60  -- base sem foto, cnpj, email
WHERE id = $auth_user_id;
```
[Source: architecture.md#9.1 DDL — tabela users, linhas 1233-1234]

**authStore — metodo updateProfile:**
```typescript
// Ja deve existir via Story 1.1 — verificar se implementado
updateProfile: async (data: Partial<User>) => {
  const { error } = await supabase
    .from('users')
    .update(data)
    .eq('id', get().user!.id);
  if (!error) {
    set((state) => ({ user: { ...state.user!, ...data } }));
  }
}
```
[Source: architecture.md#10.3 State Management — AuthStore.updateProfile]

**WatermelonDB — criacao de cliente e servico:**
- Modelos: `Client.ts` e `Service.ts` em `apps/mobile/src/database/models/`
- Sync automatica pelo SyncStore quando online
- Onboarding DEVE funcionar offline — criar entidades localmente primeiro
[Source: architecture.md#10.4 Offline Layer (WatermelonDB)]

**PostHog — rastreamento de duracao:**
- Registrar `startTime = Date.now()` ao montar `step1.tsx`
- Passar via React context de onboarding ou navigation params
- Calcular ao finalizar: `duration_seconds = Math.round((Date.now() - startTime) / 1000)`
- Propriedade `skipped_steps`: array de numeros (ex: `[2]` se pulou step 2)
[Source: architecture.md#19.1 PostHog — tabela de eventos]
