## Story

**As a** usuario do app,
**I want** completar meu perfil aos poucos conforme uso o app,
**so that** nao seja obrigado a preencher tudo no inicio.

## Acceptance Criteria

```gherkin
AC-1: Campos opcionais
GIVEN um usuario com onboarding completo
WHEN ele acessa "Meu Perfil"
THEN ve campos opcionais: foto, CNPJ/MEI, endereco, servicos detalhados, area de atuacao
AND cada campo tem indicador de preenchimento
AND barra de progresso mostra % completo

AC-2: Sugestao de completar
GIVEN um usuario com perfil < 50% completo e 3+ dias de uso
WHEN ele abre o app
THEN exibe sugestao suave (bottom sheet): "Complete seu perfil para parecer mais profissional"
AND a sugestao pode ser dispensada e nao aparece por mais 7 dias
```

### Calculo de Progresso

- [ ] **Task 9** (AC: 1, 2): Implementar calculo de profile_completion
  - Path: `packages/shared/src/utils/profileCompletion.ts`
  - Funcao pura: `calculateProfileCompletion(user: User): number`
  - Campos considerados e pesos (total = 100%):
    - name: 20% (obrigatorio, mas conta para completude)
    - phone ou email: 15% (obrigatorio, mas conta)
    - avatar_url: 25%
    - cnpj_mei: 10%
    - address: 15%
    - service_type (detalhado): 10%
    - area de atuacao: 5%
  - Retornar valor inteiro 0–100
  - Persistir no campo `profile_completion` via `authStore.updateProfile()` sempre que um campo for atualizado
  - [Source: architecture.md#4.1 User Model — campo profile_completion]

  ```typescript
  // packages/shared/src/utils/profileCompletion.ts
  export function calculateProfileCompletion(user: Partial<User>): number {
    let score = 0;
    if (user.name) score += 20;
    if (user.phone || user.email) score += 15;
    if (user.avatar_url) score += 25;
    if (user.cnpj_mei) score += 10;
    if (user.address) score += 15;
    if (user.service_type?.length) score += 10;
    if (user.area_atuacao) score += 5;
    return Math.min(score, 100);
  }
  ```
  [Source: architecture.md#4.1, architecture.md#10.3]

### Arquitetura Relevante

**User Model — campos relevantes:**
```typescript
// User model (public.users)
{
  id: string;                  // uuid, PK
  name: string;                // obrigatorio
  phone: string | null;
  email: string | null;
  city: string | null;
  service_type: string | null; // pode ser array no perfil detalhado
  avatar_url: string | null;
  cnpj_mei: string | null;
  address: object | null;      // JSON com sub-campos
  plan: 'free' | 'pro';
  role: 'gardener' | 'admin';
  onboarding_completed: boolean;
  profile_completion: number;  // 0-100
  created_at: string;
}
```
[Source: architecture.md#4.1 User Model]

**Supabase Storage — bucket `avatars`:**
- Bucket: `avatars` (configurado no Supabase)
- Path convention: `{user_id}/avatar.jpg`
- Tamanho maximo: 500KB
- Formatos aceitos: jpeg, png
- RLS: usuario so pode ler/escrever seu proprio path (`auth.uid() = user_id`)
- URL publica gerada apos upload: `supabase.storage.from('avatars').getPublicUrl(path)`
[Source: architecture.md#11.4 Storage, architecture.md#10.3]

**authStore.updateProfile():**
- Ja existe apos Story 1.3
- Assinatura esperada: `updateProfile(updates: Partial<User>): Promise<void>`
- Faz UPDATE no Supabase e sincroniza estado local do Zustand
- Apos qualquer atualizacao de campo, recalcular e persistir `profile_completion`
[Source: architecture.md#10.3 State Management]

**expo-image-picker:**
- Pacote: `expo-image-picker` (ja no Expo SDK 51+)
- Requer permissao `MEDIA_LIBRARY` (iOS: NSPhotoLibraryUsageDescription no app.json)
- `ImagePicker.launchImageLibraryAsync` para galeria
- `ImagePicker.launchCameraAsync` para camera (opcional — adicionar se escopo permitir)
[Source: architecture.md#3.1 Tech Stack]

**AsyncStorage para persistencia de sugestao:**
- Chave: `@gardengreen/profile_suggestion_dismissed_at`
- Valor: timestamp ISO string ou null
- Checar no app startup: se null ou data < (hoje - 7 dias), sugestao e elegivel
[Source: architecture.md#11.2, architecture.md#10.3]
