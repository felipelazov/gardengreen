## Story

**As a** jardineiro que usa o app em celular e tablet,
**I want** que meus dados sincronizem entre dispositivos,
**so that** sempre tenha a versao atualizada.

## Acceptance Criteria

```gherkin
AC-1: Sync entre dispositivos
GIVEN o usuario com conta logada em 2 dispositivos
WHEN ele cria um servico no device A
THEN o servico aparece no device B em < 5 segundos (com conexao ativa)

AC-2: Sync inicial
GIVEN um usuario logando em novo dispositivo
WHEN o sync inicial inicia
THEN dados sao baixados com barra de progresso
AND completa em < 30 segundos para ate 50 clientes e 500 servicos
AND o usuario pode usar o app durante o sync (dados parciais disponiveis)
```

- [ ] **Task 3** (AC: all): Criar Edge Function `sync-push` no Supabase

### Arquitetura Relevante

**WatermelonDB Sync Protocol:**
- `synchronize()` gerencia pull/push automaticamente com timestamps [Source: architecture.md#10.5]
- `lastPulledAt`: timestamp do ultimo pull bem-sucedido (null no primeiro sync)
- WatermelonDB garante consistencia local — nunca corrompe DB em caso de falha de sync

**Conflict Resolution (Last Write Wins):**
- Campo `updated_at` (timestamp UTC) em todos os registros sincronizados
- `sync-push` faz upsert apenas se `payload.updated_at >= db.updated_at`
- Dispositivo com dado mais recente sempre vence — sem merge manual
- [Source: architecture.md#10.5 Conflict Resolution]

**Supabase Realtime Channels:**
- Canal `services:{user_id}`: eventos INSERT/UPDATE/DELETE na tabela `services`
- Canal `payments:{user_id}`: eventos INSERT/UPDATE/DELETE na tabela `payments`
- Canal `clients:{user_id}`: eventos INSERT/UPDATE/DELETE na tabela `clients`
- Realtime dispara pull incremental — NAO transfere dados diretamente pelo canal
- [Source: architecture.md#10.5 Supabase Realtime]

**syncStore como fonte de verdade:**
```typescript
// Estado global de sync — consultado por toda a UI
syncStore.isOnline        // connectivity status
syncStore.isSyncing       // guard contra sync duplo
syncStore.lastSyncAt      // null = sync inicial ainda nao completou
syncStore.pendingUploads  // mudancas locais aguardando push
syncStore.syncStatus      // idle | syncing | error | success
```
[Source: architecture.md#10.5 syncStore]

**Auto-sync triggers (prioridade):**
1. Realtime event (< 1s de latencia)
2. App volta ao foreground (imediato)
3. Conexao restaurada (imediato)
4. Intervalo de 5 minutos (background)
