## Story

**As a** jardineiro trabalhando em jardim sem Wi-Fi,
**I want** que o app funcione normalmente offline,
**so that** nao dependa de internet para trabalhar.

## Acceptance Criteria

```gherkin
AC-1: Funcionamento offline
GIVEN o usuario sem conexao com internet
WHEN ele abre o app
THEN todas as telas carregam normalmente com dados locais
AND ele pode: ver agenda, cadastrar cliente, criar servico, marcar pago
AND indicador discreto mostra "Modo offline" no topo

AC-2: Sync automatico
GIVEN operacoes realizadas offline
WHEN a internet e restabelecida
THEN o sync inicia automaticamente em background
AND conflitos sao resolvidos por "ultima escrita ganha" (last write wins)
AND o usuario recebe feedback discreto: "Dados sincronizados"

AC-3: Resolucao de conflitos
GIVEN um conflito de sync (mesmo registro editado em 2 devices)
WHEN o sync detecta conflito
THEN aplica "last write wins" automaticamente
AND registra o conflito em log para auditoria
```

  ```typescript
  // apps/mobile/src/database/models/Client.ts
  import { Model } from '@nozbe/watermelondb';
  import { field, text, date, readonly, children } from '@nozbe/watermelondb/decorators';

  export class ClientModel extends Model {
    static table = 'clients';
    static associations = {
      services: { type: 'has_many' as const, foreignKey: 'client_id' },
      notes: { type: 'has_many' as const, foreignKey: 'client_id' },
      photos: { type: 'has_many' as const, foreignKey: 'client_id' },
    };
    @text('name') name!: string;
    @text('phone') phone!: string;
    @text('email') email!: string | null;
    @text('address') address!: string | null;
    @text('neighborhood') neighborhood!: string | null;
    @text('city') city!: string | null;
    @text('notes') notes!: string | null;
    @text('status') status!: string;
    @text('user_id') userId!: string;
    @readonly @date('created_at') createdAt!: Date;
    @date('updated_at') updatedAt!: Date;
    @children('services') services!: any;
    @children('notes') clientNotes!: any;
    @children('photos') photos!: any;
  }
  ```

### Arquitetura Relevante

**WatermelonDB Architecture:**
- SQLite adapter com JSI bridge para performance nativa [Source: architecture.md#3.1]
- Lazy loading e observable queries — zero overhead em listas grandes [Source: architecture.md#18.3]
- `@nozbe/watermelondb` 0.27+ — protocolo sync nativo pull/push incremental [Source: architecture.md#ADR-002]
- ADR-002: WatermelonDB escolhido sobre MMKV, AsyncStorage e Realm por SQLite adapter, lazy loading, observables e protocolo sync nativo [Source: architecture.md#ADR-002]

**Sync Protocol:**
- Sync incremental com `last_pulled_at` timestamp — apenas deltas [Source: architecture.md#10.5]
- Conflito: Last Write Wins via `updated_at` — se servidor > local, servidor vence; se local > servidor, local vence [Source: architecture.md#10.5]
- Conflitos auditados em `audit_log` para debug [Source: architecture.md#10.5]
- Fotos: apenas metadados sincronizam via WatermelonDB; upload binario por fila separada (`photoService.ts`) [Source: architecture.md#10.5]

**Auto-sync triggers (3):**
1. App foreground: `AppState 'active'`
2. Conexao restaurada: `NetInfo.addEventListener`
3. Periodic: interval de 5 minutos [Source: architecture.md#10.5]

**Performance targets:**
- Offline reads: 0ms latencia (WatermelonDB + SQLite, queries indexadas) [Source: architecture.md#18.3]
- Sync: < 10s para 100 registros (incremental sync, delta updates) [Source: architecture.md#18.3]
- WatermelonDB: lazy loading, observable queries, batch operations [Source: architecture.md#18.3]
